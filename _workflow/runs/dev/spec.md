## 2. Original Request
- Raw user request: Replace SMTP/Nodemailer order notifications with Telegram Bot API notifications.
- Normalized request: Implement a tested native-fetch Telegram notifier, require Telegram credentials, preserve notifier injection/failure isolation, and remove SMTP/Nodemailer dependencies and references.
- Source prompt / request reference: `_workflow/runs/dev/request.md`

## 3. Questions And Answers

- Answers received: Preserve order number, customer name, total, and phone as plain text; throw on HTTP/API failure.
- Questions skipped: None.

## 4. Problem Definition
- Problem being solved: Order notifications currently depend on SMTP and Nodemailer rather than Telegram.
- Why it matters: Operations require Telegram-only delivery with fewer provider-specific dependencies.
- Current pain point: SMTP variables, dependency, code, tests, and docs remain coupled to email.
- Expected value: Direct Telegram notifications using the Node runtime only.

## 5. Current State Analysis
- Existing behavior: `createNotifier(config)` returns an email sender; absent SMTP host silently disables notifications.
- Existing architecture/components: `orders.js` injects the notifier; `orderService.js` invokes it asynchronously and catches/logs failures.
- Existing files/modules likely involved: notification service, env config, env example, package manifests, server tests, operations/project/architecture/verification docs.
- Existing data flow: Persist order, invoke `deps.notify(order)`, log rejection without rollback.
- Existing API/UI/CLI/workflow behavior: No notification endpoint or UI; notification is an internal side effect.
- Existing tests or verification coverage: Order-service isolation exists indirectly; no focused notifier test exists; operations docs test expects SMTP variables.

## 6. Desired End State
- Expected final behavior: Every created order triggers a Telegram `sendMessage` request.
- User-facing outcome: Staff receive order number, customer, total, and phone in the configured chat.
- Developer-facing outcome: No Nodemailer/SMTP configuration; notifier is testable through mocked `fetch`.
- System/workflow outcome: Failed notifications reject and are caught/logged by the unchanged order service.
- Backward compatibility expectations: Preserve `createNotifier(config)` and `notify(order)` contracts; SMTP environments intentionally cease to work.

## 7. Scope
- In scope: Telegram notifier, strict config, env example, dependency removal, lockfile, tests, notification-related docs.
- Out of scope: Bot creation, chat setup automation, retries, queues, dashboards, message localization, HTML/Markdown formatting.
- Non-goals: Altering order creation, route composition, persistence, or failure logging.
- Explicit boundaries: No Axios and no frontend changes.

## 8. Users And Use Cases
- Primary users: Restaurant staff monitoring the configured Telegram chat.
- Secondary users: Operators configuring deployment secrets; developers maintaining order flow.
- Main use cases: Receive a concise notification after a successful order.
- Edge use cases: Telegram network failure, non-2xx response, or API-level rejection.

## 9. Functional Requirements
- Required behaviors: POST JSON to `/bot<TOKEN>/sendMessage`; include configured `chat_id`; construct confirmed plain text; reject all unsuccessful outcomes.
- Inputs: `config.telegram.botToken`, `config.telegram.chatId`, and persisted order.
- Outputs: Resolving notifier promise on Telegram success; rejected promise with non-secret diagnostic on failure.
- State changes: Telegram receives a message; application state remains unchanged.
- Error states: Network rejection, non-2xx HTTP, invalid JSON, and `ok !== true` response.
- Permissions/auth expectations: Bot token remains environment-only and must not appear in logs/errors generated by application code.

## 10. Non-Functional Requirements
- Performance expectations: One non-blocking outbound request per new order.
- Reliability expectations: Notification failure never rolls back an order; failures remain observable through current logging.
- Security/privacy expectations: HTTPS Telegram endpoint; do not log token; send only confirmed fields.
- Accessibility expectations: Not applicable.
- Maintainability expectations: Small CommonJS module with injectable/global fetch mockability and focused tests.
- DX expectations: Startup config validation names missing Telegram variables clearly.

## 18. Edge Cases And Failure Modes
- Edge cases: Special characters in names/phone, negative or absent totals from malformed internal input, large Telegram response bodies.
- Failure modes: DNS/network errors, 401 invalid token, 400 invalid chat, 429 rate limit, non-JSON response, `ok: false` payload.
- Regression risks: Fetch mock leakage, token exposure in URL-bearing errors, startup failures in tests lacking new required env values.
- Recovery expectations: Existing catch logs a sanitized error; staff use the dashboard as authoritative queue.

## 19. Risks And Mitigations
- Technical risks: Treating HTTP 200 as success without checking Telegram payload; mitigate with explicit response validation/tests.
- Product/UX risks: Telegram message becomes unavailable during provider outage; retain dashboard authority and logging.
- Security risks: Bot token leakage; never include endpoint/token in application errors.
- Scope risks: Expanding into retry infrastructure; explicitly excluded.
- Mitigation plan: Focused contracts, repository-wide reference search, full verification.

## 20. Assumptions
- Explicit assumptions: Node 22 global fetch is available; Telegram supports JSON `sendMessage`; GBP/pence representation remains unchanged.
- Confidence level: High.
- What to revisit if assumptions are wrong: Add a runtime-compatible fetch strategy only if deployment is not Node 22; this would require approval.
